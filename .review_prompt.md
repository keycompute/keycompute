### Security Principles (NON-NEGOTIABLE)
1. **Permission-based access control**: AuthContext's `has_permission()` is the sole gatekeeper — never check `role` strings directly for authorization decisions. API Key authentication MUST only grant `UseApi` permission even if the user role is `"admin"`. JWT authentication builds permissions from role + AuthType::Jwt.
2. **API Key security**: API keys (`sk-` format) MUST be hashed (SHA256) before storage. Never store or log plaintext API keys. The `decrypt_upstream_api_key()` in routing MUST use AES-256-GCM decryption when `global_crypto()` is set, falling back to plaintext only when explicitly unconfigured (development mode).
3. **Password security**: Passwords MUST be hashed with Argon2id (64MB memory, 3 iterations, 4 parallelism) using random salts. Never use raw bcrypt or custom hashing. The `PasswordHasher` module in `keycompute-auth/src/password/hash.rs` is the sole entry point.
4. **Token invalidation**: JWT `token_version` tracking in the database MUST be used for password changes, role changes, and account suspension. The auth middleware MUST verify the JWT's token_version against the DB on every authenticated request.
5. **Rate limiting fail-closed**: Rate limit check failure (Redis unavailable) MUST deny the request, not allow it through. The `check_and_record_with_config` method must propagate errors. For memory backend, atomic check-and-record with rollback MUST be used (increment → check → decrement on overflow).
6. **Error information hiding**: Internal error details MUST NOT be exposed to the client via API responses. Convert internal errors into generic messages via `ApiError::Internal`. Node identity mismatch errors MUST hide the specific expected/actual IDs.
7. **Node gateway authentication**: Node registration tokens are single-use and consumed atomically in a transaction. HMAC signature verification is required for registration. Session tokens are used for heartbeat/poll/complete operations.
8. **Default credential rejection**: In non-development environments, the server MUST refuse to start if `jwt_secret`, `encryption_key`, or admin credentials use default values.

### Review Rules
1. Only report issues supported by evidence in the diff or the current codebase. Do not speculate about code you cannot see. If something is uncertain, label it as a hypothesis.
2. Prioritize (in descending order):

   - Security (auth bypass, privilege escalation, injection, secret exposure, SSRF)
   - Error handling (unhandled errors, incorrect error mapping, information leaks)
   - Edge cases (empty states, boundary values, concurrent access, DB constraint violations, decimal precision)
   - Maintainability (code duplication, overly complex logic, unclear naming, crate dependency direction)
   - Missing tests (unit tests for services, integration tests for handlers)
3. Distinguish clearly between **Blocking Issues** (must fix) and **Non-blocking Suggestions** (nice to have).
4. If no obvious blocking issue is found, say so clearly — do not invent issues.
5. Keep the review concise, concrete, and focused on the files under review. Reference filenames with line numbers.
6. **Diff truncation awareness**: The diff you receive may be truncated at file or total character limits. When a file's patch ends with `...[truncated]`, note that you have only seen part of the change and avoid definitive judgments about the omitted portion.
7. Pay special attention to:
   - Transaction boundaries in billing and distribution (are reads/writes within the same transaction?)
   - Decimal ↔ BigDecimal conversion in billing (string-bridge pattern is critical — any silent fallback is a blocking issue)
   - Argon2id memory/time parameters (are they appropriate for the use case?)

   - Node gateway task submission (are duplicate submissions detected via request_hash? is idempotency guaranteed?)
   - DbRouter read-write separation (are write operations directed to the write pool? read-only queries to the read pool?)
   - JWT validation (is `iss` checked? is `exp` enforced? are validation errors generic?)
   - Distribution calculation (is decimal precision maintained? are edge cases like zero amounts handled?)
   - WASM compatibility (are native-only APIs gated behind cfg(target_arch)?)
8. When reviewing configuration changes (config.toml, Cargo.toml, Dockerfile, docker-compose.yml):

## Appendix: KeyCompute Key Code Patterns

### Project-specific patterns to check for consistency:

1. **AuthContext permission model**: Never check `role` directly for authorization — always use `AuthContext::has_permission()`. API Key auth (`AuthType::ApiKey`) MUST only grant `UseApi` permission regardless of role. JWT auth (`AuthType::Jwt`) grants permissions based on role.

2. **API key lifecycle**: Generate with `ProduceAiKeyValidator::generate_key()` or `generate_key_with_prefix()`. Hash with SHA256 (`hash_key()`) before storage. Validate format with `is_valid_format()` (sk- prefix, 51 chars standard or sk-{prefix}-{random} format). On auth, query by hash, never by plaintext key.

3. **Upstream API key encryption**: AES-256-GCM via `ApiKeyCrypto`. Use `encrypt_api_key()` / `decrypt_api_key()` helpers. Global key set once at startup via `set_global_crypto()`. When unconfigured (dev mode), plaintext fallback is acceptable but must emit a clear warning.

4. **Password hashing**: Use `PasswordHasher` (Argon2id, 64MB memory, 3 iterations, 4 parallelism). Passwords MUST NOT be stored in plaintext. Verification code hashing for email flow also uses Argon2id.

5. **JWT validation**: Use `JwtValidator::validate()` with HS256 algorithm. Always validate `iss`. Never expose specific failure reasons in API responses — use generic "Invalid or expired token" messages.

6. **Rate limiting atomicity**: Use `RateLimitService::check_and_record_with_config()` for atomic check-and-record. Memory backend uses increment → check → decrement-on-overflow pattern. Redis backend uses Lua scripts. Never use separate check-then-record patterns.

7. **Billing architecture**: `BillingService::finalize_and_trigger_distribution()` is the main settlement entry point. Balance deduction, distribution, and node tipping are fire-and-forget (failures MUST NOT affect the primary execution result). Decimal → BigDecimal conversion must use string bridge pattern.

8. **Routing engine**: Two-layer routing (Layer1: provider scoring, Layer2: tenant-scoped account selection). Routing is read-only and side-effect-free. Provider health and account cooldown states are read via in-memory stores. Node routing uses `node:` model prefix with `NodeCapabilityIndex` trait.

9. **Provider adapter pattern**: All LLM providers implement `ProviderAdapter` trait. `stream_chat()` returns `StreamBox` (Pin<Box<dyn Stream>>). Non-streaming `chat()` has a default implementation via streaming. See `keycompute-provider-trait` for the contract.

10. **Retry + backoff**: Use `RetryPolicy` (default: max_retries=3, initial_backoff_ms=100, multiplier=2.0, max_backoff_ms=10000). Do NOT retry on `AuthError` or `RateLimitExceeded`. Use `RetryState` to track attempts and backoff.

11. **Node gateway task idempotency**: Task submissions use `request_hash` (SHA256 of canonical JSON of task_id + lease_id + result) for duplicate detection. Registration tokens are single-use, consumed atomically in a transaction. Session tokens validated on every heartbeat/poll/complete.

12. **Error mapping chain**: `keycompute_types::KeyComputeError` → `keycompute_server::ApiError` → Axum `IntoResponse`. Node task conflict errors must map to 409, client errors to 400, provider errors to 502. Internal error details MUST NOT leak to the client.

13. **DbRouter read-write separation**: Write operations must go through the write pool. Read-only queries should use the read pool when available. `DbRouter::single()` is used for single-DB setups. The `DbRouter` type should be used via `Arc<DbRouter>` across service boundaries.

14. **Service construction pattern**: Services follow a builder pattern with optional database connections. Example: `BillingService::new()` (no DB) → `BillingService::with_pool(pool)` (with DB). The `has_pool()` method is used during startup validation. Services in tests can be constructed without a database.

15. **Observability**: Use `tracing` for structured logging with request_id context. Provider health tracking uses `ProviderHealthStore` with sliding window. Account cooldown uses `AccountStateStore` with TTL-based eviction.
